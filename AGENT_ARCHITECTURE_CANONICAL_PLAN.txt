EXECUTIVE POSITION

The current system is no longer at the "make basic RAG work" stage.
It already has:
- stable full-corpus ingestion
- Postgres as the main system of record
- Chroma as the vector retrieval layer
- KG extraction and repair
- multimodal assets
- user-scoped sensors
- hybrid lexical+dense retrieval
- KG evidence in prompt and grounding
- exact-query embedding caching
- a merged React admin direction

The next architecture goal is not "more features at random."
The next architecture goal is:

1. make retrieval structurally stronger than plain hybrid RAG
2. make memory disciplined and evidence-aware
3. make graph retrieval real without prematurely adding a graph database
4. make maintenance and serving operations as safe as fresh ingest
5. make the runtime observable, replayable, and budget-governed

CANONICAL DECISIONS

The following decisions are now the default architecture unless there is a concrete reason to override them.

1. Postgres remains the system of record.
- Documents, chunks, validations, KG, sessions, memory, traces, evals, review state, caches, and audit data stay in Postgres.

2. Chroma remains a retrieval-serving layer, not the source of truth.
- Chroma exists to serve retrieval fast.
- Chroma does not define truth.
- Retrieval-visible writes must be staged before they are promoted live.

3. The agent remains single-orchestrator, not a manager-of-specialists topology.
- The system should use one bounded serving agent with well-defined internal stages.
- Multi-agent runtime orchestration is not the next priority.

4. Chunk and evidence grounding remain the authority.
- KG, sensors, summaries, and memory help organize evidence.
- Final answers must still be grounded in chunk text, trusted asset text, sensor rows, or explicit KG evidence rows that themselves point back to source evidence.

5. Graph-style retrieval should be added before any graph database.
- Build graph retrieval on top of the existing KG in Postgres first.
- Add a graph DB only if real workloads prove Postgres traversal is the limiting factor.

6. Retrieval-visible maintenance paths must become staged.
- Fresh ingest is no longer the only flow with staged publish semantics.
- Reindex, revalidate, repair, and review-driven accepts should converge toward the same visibility rules.

7. Memory writes must be contract-based.
- The system must not write free-form "facts" into memory unless the write is clearly sourced, typed, and bounded.

8. All external credentials should move toward vault-backed access.
- Runtime secrets should come from 1Claw or a similar just-in-time secret source, not broad plaintext database storage.

WHAT TO KEEP

The following parts of the current architecture are directionally correct and should be kept.

- Postgres as source of truth
- Chroma as vector retrieval layer
- ontology-aware KG extraction
- user-scoped places, hives, sensors, and readings
- exact-query embedding cache
- router-pattern cache
- hybrid lexical+dense retrieval
- KG evidence as first-class context
- deterministic evidence-summary fallback when the answer model fails
- React as the primary frontend stack

WHAT IS STILL MISSING

The system is good enough to run, but it is not yet the best version of the architecture.
The main missing pieces are:

1. hierarchical retrieval
- The system still retrieves mostly at the chunk level first.
- It needs document, section, and chunk layers.

2. graph-style retrieval
- KG is attached after retrieval, but the system does not yet retrieve by graph path or graph neighborhood.

3. disciplined memory
- Session/profile summaries exist, but the memory model still needs stronger typing, evidence policies, and decay rules.

4. richer evaluation
- The system needs a deeper evaluation program for explanation, comparison, visual, sensor, and graph-assisted questions.

5. staged maintenance operations
- Fresh ingest is safer than some maintenance paths.
- That asymmetry should be reduced.

6. stronger runtime governance
- Better secret handling
- better cost controls
- better request budgets
- better operator observability

TARGET ARCHITECTURE

The architecture should be thought of as six planes.

1. DATA PLANE

Stores:
- Postgres
- Chroma
- filesystem-backed page assets
- auth Postgres

Responsibilities:
- durable truth
- provenance
- audit
- cache rows
- eval datasets
- operator history

Rules:
- Postgres tables own truth
- Chroma owns search acceleration only
- every retrieval-visible record has a stable source row in Postgres
- every graph assertion has evidence provenance
- every cache row is scoped by model/provider/version identity

2. INGESTION PLANE

Responsibilities:
- source registration
- parsing
- normalization
- chunk construction
- chunk validation
- embedding
- vector staging/publish
- KG extraction
- KG validation
- review and repair
- asset linking
- replay and recovery

Rules:
- stages are durable and replayable
- failures are visible
- maintenance writes follow the same safety contract as fresh ingest
- no stage should destroy the last known good retrieval surface before replacement is ready

3. RETRIEVAL PLANE

Responsibilities:
- request normalization
- auth and tenant/resource scope enforcement
- route planning
- candidate generation
- candidate reranking
- graph expansion
- sensor context expansion
- bundle budgeting
- prompt construction
- grounding and abstention

Rules:
- retrieval should be hybrid by default
- graph expansion should be query-type aware
- visual retrieval should be conditional, not unconditional
- sensor retrieval should be explicit or strongly inferred, not accidental

4. MEMORY PLANE

Responsibilities:
- session continuity
- profile/personalization
- open-thread tracking
- verified user facts
- preferred source scope
- operator review feedback

Rules:
- memories are typed
- fact memories require evidence or explicit user self-report
- preferences are not the same as facts
- unresolved thread memory is not the same as domain knowledge
- memory must be compact, inspectable, and deletable

5. CONTROL PLANE

Responsibilities:
- admin UI
- operator inspection
- eval execution
- repair flows
- model/config management
- auth and permissions
- runtime controls
- SQL and system views

Rules:
- one primary React admin surface
- legacy UI only as temporary fallback
- every destructive action should be inspectable and auditable

6. RUNTIME / INFRA PLANE

Responsibilities:
- containers
- health checks
- secret delivery
- queues or detached jobs
- structured logging
- backup/restore
- resource isolation

Rules:
- compose defines the local operational baseline
- services are separated by concern
- long-running work should not depend on the API container lifecycle
- secrets should be fetched just in time and rotated cleanly

TARGET REQUEST FLOW

The serving path should become this:

1. Authenticate and authorize.
- Resolve auth user
- resolve tenant
- resolve permitted documents, places, hives, and sensors
- reject oversized or invalid requests early

2. Normalize and classify.
- Normalize the question
- classify question type
- detect whether the question is general, sensor, or mixed
- detect whether graph-style retrieval is likely useful

3. Select a retrieval profile.
- fact
- definition
- source lookup
- explanation
- procedure
- comparison
- visual
- sensor
- mixed corpus + sensor

Each profile should set:
- lexical search width
- dense search width
- whether assets are searched in pass one
- whether graph expansion is allowed
- whether neighbor expansion is allowed
- prompt budget quotas by evidence type

4. Generate initial candidates.
- lexical chunk candidates from Postgres FTS / trigram
- dense chunk candidates from Chroma
- lexical asset candidates when appropriate
- dense asset candidates when appropriate
- sensor candidates when sensor mode applies
- seed KG entities from the query and top chunk candidates

5. Rerank candidates.
- dense score
- lexical score
- section/title overlap
- exact phrase overlap
- ontology metadata overlap
- document and section diversity
- verified KG density

6. Build graph expansion candidates.
- only for supported query types
- seed from the question and top retrieved chunks
- expand through a small local neighborhood
- rank assertions and short paths
- attach evidence rows and supporting chunks

7. Assemble the context bundle.
- selected chunks
- selected assets
- selected sensor rows or aggregates
- selected assertions
- selected entities
- selected KG evidence
- selected graph paths
- session memory
- profile memory

8. Generate the answer.
- strict response schema
- allowed ids only
- no uncited answer claims
- confidence and abstention are mandatory

9. Ground and verify.
- claim verifier when enabled
- lexical and structural fallback
- evidence-id validation
- citation validation
- abstain on failure

10. Persist the trace.
- query run
- selected sources
- selected evidence
- selected graph paths
- response metadata
- budgets and latency metrics

11. Update memory through typed contracts.
- preferences
- explicit user facts
- open threads
- stable verified facts

RETRIEVAL ARCHITECTURE: WHAT TO BUILD

The retrieval stack should be improved in this order.

A. HIERARCHICAL RETRIEVAL

Current weakness:
- retrieval is still chunk-first with later expansion

Target:
- document synopsis layer
- section synopsis layer
- chunk layer

How:
- create document summary nodes and section summary nodes during ingest
- embed those summaries separately
- retrieve document and section candidates first for broad queries
- restrict chunk search inside the surviving documents/sections

Why:
- better precision on long books
- better comparison answers
- lower prompt noise
- better control over document spread

B. TRUE HYBRID RETRIEVAL

Current state:
- lexical + dense candidate merging already exists

Target:
- keep hybrid retrieval as the default
- improve lexical indexing quality and diagnostics

How:
- add explicit FTS/trigram indexes for chunk text, titles, section labels, and asset searchable text
- store match-source diagnostics per result
- expose lexical vs dense contributions in admin replay tools

Why:
- rare terms
- OCR-heavy strings
- exact titles
- citations with rare domain names

C. CONDITIONAL ASSET RETRIEVAL

Current issue:
- asset retrieval has improved, but the architecture should formalize when it is allowed

Target:
- no pass-one asset search for plain text-first questions unless the route demands it

How:
- visual retrieval profile
- asset expansion only when:
  - the route is visual
  - chunk links imply relevant assets
  - or first-pass grounding is weak

D. GRAPH-STYLE RETRIEVAL

This is the next major architecture improvement.

Definition:
- retrieve connected facts, not just semantically similar chunks

How it should work:

1. Seed entities.
- From the normalized question
- from the top chunk candidates
- from exact lexical entity matches

2. Expand a local neighborhood.
- max 1 hop initially
- later 2 hops for selected query types
- predicate whitelist by query type
- minimum confidence and evidence thresholds

3. Build ranked path objects.
- assertion sequence
- involved entities
- supporting evidence ids
- supporting chunk ids
- path score

4. Include compact paths in the prompt.
- not raw graph dumps
- only a few short chains

5. Require grounding back to evidence.
- graph paths are useful only if they point to evidence

Where to build it:
- Postgres first
- not a graph DB yet

When to use it:
- explanation
- procedure
- comparison
- causal or dependency questions

What to avoid:
- unrestricted neighborhood dumps
- using KG without source evidence
- using graph retrieval on every question

E. RERANKING IMPROVEMENTS

Add a second-stage reranker for hard questions.

Target:
- rerank top 20-40 chunk candidates with a stronger model or cross-encoder style signal

Use only when:
- the question is broad or complex
- the first pass is ambiguous
- the query type is explanation, procedure, or comparison

Why:
- better final context selection
- less prompt waste

F. RESPONSE CACHING

Not a first move, but worth planning.

Target:
- safe read-only answer caching for repeated identical questions under the same corpus snapshot and runtime config

Rules:
- scope by tenant
- scope by corpus snapshot/version
- scope by query mode
- invalidate on corpus mutation or config version change

Do not do this before:
- retrieval and grounding regressions are stable

KG ARCHITECTURE: WHAT TO BUILD

A. KEEP THE CURRENT MODEL

The KG should continue to be:
- ontology-constrained
- provenance-backed
- evidence-linked
- stored in Postgres

B. IMPROVE CANONICALIZATION

Targets:
- stronger entity alias handling
- explicit synonym tables
- predicate-level quality metrics
- stable resolution rules by ontology version

How:
- maintain canonical entity tables plus alias tables
- store resolution decisions and confidence
- record ontology version on extraction and validation

C. ADD GRAPH QUALITY METRICS

Measure:
- assertion count by predicate
- evidence coverage per assertion
- orphan entities
- orphan assertions
- high-degree generic entities
- quarantine/review rates by predicate and document class

These metrics are required before graph-style retrieval rollout.

D. ADD PATH OBJECTS AS A DERIVED LAYER

Do not create a separate database yet.
Create derived query-time path objects:
- seeded entity ids
- traversed assertion ids
- evidence ids
- chunk ids
- path relevance score
- path explanation text

E. ADD GRAPH REPAIR TOOLING

Operators need:
- replay KG for selected documents/chunks
- inspect bad predicates
- inspect alias merges
- inspect path candidates

SENSOR ARCHITECTURE: WHAT TO BUILD

The sensor system should become a first-class retrieval modality, not just an appended context block.

1. Add explicit sensor query parsing.
- identify sensor ids, place ids, hive ids, measurement types, and time windows

2. Add canonical aggregations.
- latest
- average over window
- min/max
- trend
- anomaly flags

3. Add unit normalization.
- store canonical units and conversion metadata

4. Add mixed-mode retrieval.
- Questions like "Could temperature stress explain this colony behavior?" should combine:
  - sensor evidence
  - corpus evidence
  - graph support if needed

5. Keep sensor citations deterministic.
- row ids
- time windows
- aggregate method
- source sensor and hive/place identity

MEMORY ARCHITECTURE: WHAT TO BUILD

Current memory should be restructured into four types.

1. SESSION OPERATING MEMORY
- active goal
- current thread
- unresolved asks
- recent decisions

2. PROFILE / PREFERENCE MEMORY
- communication style
- preferred answer depth
- domain experience level
- known preferred documents or scopes

3. VERIFIED FACT MEMORY
- explicit user-provided facts
- durable user facts derived from repeated confirmation
- facts must have:
  - source type
  - confidence
  - created_at
  - expiry or review policy

4. RETRIEVAL MEMORY
- previously useful documents
- preferred sections for recurring questions
- recent high-quality evidence bundles

MEMORY WRITE POLICY

The system should not write memory freely.

Rules:
- explicit user self-report can create a user fact
- model inference alone should not create a durable fact
- evidence-backed domain facts should stay in corpus/KG, not profile memory
- unresolved threads should expire
- preferences can be updated, but facts require stronger gates

MEMORY READ POLICY

Prompt injection from memory is a real risk.
Memory injection should therefore be:
- typed
- compact
- size-capped
- inspected in admin
- removable by type

INGESTION ARCHITECTURE: WHAT TO BUILD NEXT

The core ingest works, but the architecture should be tightened further.

1. Extend staged publish to all retrieval-visible maintenance flows.
- reindex
- revalidate
- review-driven accept/reject
- repair operations

2. Add synopsis generation during ingest.
- document synopsis
- section synopsis
- maybe table/figure summaries when useful

3. Add lexical index preparation during ingest.
- normalized searchable text
- trigram/FTS-ready fields
- title and heading terms

4. Add graph-readiness metrics during ingest completion.
- graph coverage
- evidence coverage
- quarantine counts
- entity resolution quality

5. Add corpus snapshot/versioning.
- every completed corpus state should have a version id
- retrieval traces must record the corpus snapshot used

6. Add immutable prompt/model version recording everywhere.
- answer model
- router model
- KG model
- reviewer model
- verifier model

SECURITY, SECRET, AND RUNTIME PLAN

A. SECRET HANDLING

Use the 1Claw direction for the target state.

Target:
- model keys
- DB credentials
- admin/system credentials
- runtime secret encryption keys

should be obtained from a vault/env bundle, not broadly persisted in application tables.

Rules:
- metadata-first secret discovery
- just-in-time reads only
- no raw secret values in logs, prompts, or persisted operator text
- immediate rotate-and-store flow after key rotation

B. API SAFETY

Every public and admin endpoint should converge on:
- deny by default
- strict request validation
- bounded input sizes
- request-type rate limits
- object-level auth checks
- durable audit logging

C. RUNTIME TOPOLOGY

Keep the current Docker direction, but harden it.

Baseline services:
- api
- worker
- postgres
- auth-postgres
- chroma

Later optional services:
- queue
- dedicated eval worker
- dedicated repair/replay worker
- metrics/log collector

Rules:
- long-running ingest or replay cannot depend on API container uptime
- every worker type should have a clear concern boundary
- health checks should distinguish:
  - process up
  - dependencies reachable
  - model egress functioning

D. BACKUP AND RESTORE

Required:
- Postgres backup
- auth-postgres backup
- Chroma persistence backup
- restore drill
- corpus snapshot and restore documentation

E. COST GOVERNANCE

Required controls:
- per-model usage dashboards
- per-request token metrics
- question-type latency and cost metrics
- daily budget alarms
- circuit breakers on provider failure spikes

This is mandatory.

Build five eval tracks.

1. RETRIEVAL EVAL
- chunk recall
- asset recall
- lexical vs dense contribution
- document concentration
- graph expansion quality

2. ANSWER EVAL
- grounded answer correctness
- abstention correctness
- citation precision
- unsupported claim rate

3. SENSOR EVAL
- time-window correctness
- aggregate correctness
- mixed sensor + corpus correctness

4. MEMORY EVAL
- useful continuity
- preference recall
- fact memory correctness
- unwanted stale carryover

5. OPERATOR / MAINTENANCE EVAL
- repair safety
- revalidation safety

Every major rollout should have:
- baseline score
- candidate score
- regression diff
- explicit go/no-go decision

GRAPH DATABASE DECISION

Do not add a graph database yet.

Use Postgres first for:
- seeded entity lookup
- 1-hop traversal
- constrained 2-hop traversal
- path construction
- evidence joins

Reconsider a graph DB only if these become true:
- graph-style retrieval is already proven valuable
- multi-hop traversal is common in production
- path ranking queries in Postgres become slow or awkward
- interactive graph exploration becomes a real operator or product feature
- graph algorithms become part of the serving path

If added later:
- the graph DB should begin as a derived read model
- Postgres should still own the canonical write model and provenance

THINGS THE ARCHITECTURE SHOULD EXPLICITLY AVOID

1. Do not let user chat mutate the corpus.
2. Do not let memory become a free-form fact store.
3. Do not let KG assertions outrank source evidence.
4. Do not let graph retrieval expand without tight limits.
5. Do not add a graph database before graph-style retrieval proves its value.
6. Do not unify router cache and embedding cache.
7. Do not cache answers without corpus snapshot scoping.
8. Do not make the system depend on a single UI surface without replay and inspection tooling.
9. Do not keep secrets as broad plaintext runtime overrides if a vault-backed path is available.
10. Do not add multi-agent orchestration before the single-agent retrieval stack is structurally strong.

PRIORITIZED EXECUTION ORDER

PHASE 0 - BASELINE FREEZE

Goal:
- freeze the current system into a measurable baseline

Build:
- corpus snapshot id
- model and prompt registry
- retrieval eval datasets by question type
- graph quality metrics
- memory type definitions
- cost dashboard basics

Exit criteria:
- every query run records corpus snapshot, model versions, and prompt version
- baseline eval scores exist

PHASE 1 - RETRIEVAL EXCELLENCE

Goal:
- make the core retrieval stack strong before graph expansion

Build:
- hierarchical document/section/chunk retrieval
- better lexical indexes
- conditional asset retrieval formalization
- optional second-stage reranker
- richer retrieval diagnostics in admin

Exit criteria:
- better retrieval precision on explanation/comparison questions
- lower prompt noise
- no regression on fact/source-lookup questions

PHASE 2 - GRAPH-STYLE RETRIEVAL

Goal:
- use the KG as a real retrieval surface, not just attached support

Build:
- entity seeding
- predicate-aware local expansion
- ranked path objects
- graph bundle prompt section
- graph path admin inspector

Exit criteria:
- explanation/procedure/comparison eval scores improve
- unsupported-claim rate does not rise
- graph bundle remains small and evidence-backed

PHASE 3 - MEMORY REBUILD

Goal:
- make memory useful and safe instead of loosely summarized

Build:
- typed memory schema
- write policies
- expiry and review policies
- memory inspector
- better prompt injection of only relevant memory slices

Exit criteria:
- continuity improves without factual contamination
- stale or incorrect memory is easy to inspect and delete

PHASE 4 - MAINTENANCE SAFETY AND SECRET HARDENING

Goal:
- make all maintenance operations and runtime secrets production-safe

Build:
- staged publish for maintenance flows
- 1Claw-backed secret handling path
- audit improvements
- restore drills
- stronger runtime budgets and circuit breakers

Exit criteria:
- repair/reindex/revalidate are no longer materially riskier than fresh ingest
- secret exposure surface is reduced

PHASE 5 - CONTROL PLANE COMPLETION

Goal:
- make the admin UI the real operator cockpit

Build:
- full React admin parity
- graph inspector
- memory inspector
- replay diffing
- corpus snapshot navigation
- budget and provider dashboards

Exit criteria:
- legacy admin no longer needed for daily operations

ACCEPTANCE STANDARD FOR "GOOD ENOUGH"

The architecture should be considered in a strong state when all of these are true:

1. Retrieval is hierarchical, hybrid, and graph-assisted where appropriate.
2. Every answer trace can explain exactly why each source was included.
3. Graph-style retrieval improves targeted question classes without harming grounding.
4. Memory is typed, inspectable, and hard to poison.
5. Maintenance flows are staged, observable, and replayable.
6. Secrets are not broadly persisted in plaintext operational paths.
7. Admin provides one primary operator surface.
8. Evaluation is a gating system, not an afterthought.

FINAL ARCHITECTURE DIRECTION IN ONE SENTENCE

The right end state is a single grounded serving agent built on Postgres truth, Chroma acceleration, evidence-backed KG expansion, disciplined typed memory, sensor-aware retrieval, staged maintenance writes, vault-backed secrets, and a fully inspectable React control plane.
