Purpose

System interpretation

This repository is a deterministic AI chat and RAG system, not an autonomous multi-agent runtime.

The serving path is a single modular monolith that owns:
- HTTP APIs
- auth/session handling
- deterministic agent orchestration
- retrieval
- memory
- admin/operator behavior
- hive workspace behavior

The offline path is a detached worker/job plane that owns:
- ingestion
- reindexing
- repair
- replay
- KG enrichment
- multimodal/source preparation
- maintenance and backfill jobs

Hard constraints

- Do not turn the runtime into a multi-agent system.
- Do not add manager/worker agents or unconstrained tool loops.
- Do not split retrieval into a separate service.
- Do not add internal HTTP calls between modules in the same product.
- Preserve deterministic behavior, provenance, grounding, and predictable latency.
- Preserve behavior unless a bug fix or explicit policy change requires otherwise.

Source-of-truth model

- Postgres is the canonical source of truth for documents, chunks, sessions, memory, traces, runtime config, and review state.
- Chroma is derived retrieval state.
- KG/enrichment is derived state with provenance back to relational rows.
- Caches are disposable.

Retrieval-visible state rules

- Retrieval-visible vector writes must be staged before they become live.
- Publish is the only offline boundary that should make staged document state retrieval-visible.
- Answer generation must ground on chunk text, trusted asset text, trusted sensor rows, or KG evidence rows that themselves resolve back to source evidence.

Primary bounded contexts

`http_api/`
- Owns HTTP route declarations, request/response DTOs, app factory wiring, and HTTP-edge auth/session helpers.
- Must not own business workflows, SQL, prompt logic, retrieval logic, or offline execution.

`agent_runtime/`
- Owns the deterministic online pipeline:
  - request guard integration
  - session/profile resolution
  - planner
  - evidence packing
  - prompt assembly
  - model invocation
  - verification
  - trace and memory coordination
- Must remain a single orchestrator/facade, not a multi-agent system.

`retrieval/`
- Owns lexical, dense, rerank, and KG-assisted retrieval mechanics.
- Must not own prompt construction, memory policy, or route behavior.

`storage/`
- Owns persistence by bounded context.
- Must not own route behavior, prompt behavior, or schema mutation in constructors.

`admin/`
- Owns operator workflows and policy that are more than HTTP translation.
- Must not own route declarations or low-level SQL.

`offline_pipeline/`
- Owns stage contracts, runner behavior, and offline stage/pipeline composition.
- Must not import HTTP modules or chat prompt modules.

`hive_domain/`
- Owns hive/beekeeping-specific domain logic such as telemetry interpretation and report semantics.
- Must not absorb generic platform concerns.

`frontend/src/features/*`
- Page components own layout and UI orchestration.
- Feature hooks, sections, and models own client-side feature logic.
- Page components must not become hidden application cores.

Deterministic agent runtime stages

The online runtime should stay explicitly staged:
1. request guard
2. conversation/session/profile resolution
3. planning and query-mode selection
4. retrieval orchestration
5. evidence packing and budgeting
6. prompt construction
7. model invocation
8. answer verification and grounding checks
9. trace persistence and memory updates

Memory rules

Memory is a first-class subsystem and must remain:
- typed
- bounded
- inspectable
- removable

Typed memory classes:
- session continuity
- user preferences
- user self-reported facts
- open threads/tasks
- operator corrections

Rules:
- retrieved corpus facts are not memory
- arbitrary model output must not become durable memory without explicit policy
- shared profile memory must not contaminate session-scoped chat continuity

API rules

- Route handlers stay thin.
- Routes call application services or orchestrators.
- `api.py` is only a compatibility entrypoint and composition surface.
- Control-plane auth stays at the HTTP edge.
- Long-running ingestion must not execute inline in request handlers.

Storage rules

- Bounded stores own SQL/query logic for their context.
- `repository.py` is a compatibility facade until the remaining persistence areas are extracted.
- No new bounded-store behavior should be added directly to `repository.py`.
- Repository construction must stay free of hidden schema mutation. Compatibility/bootstrap work belongs to explicit startup/bootstrap code.

Offline pipeline rules

- Normal ingestion, repair, replay, and reindex are offline concerns.
- Worker entrypoints may call a pipeline runner/facade, but request handlers must only enqueue or trigger jobs.
- Publish remains the retrieval-visibility boundary.

Frontend rules

- Chat pages stay UI-focused.
- Admin pages stay UI-focused.
- Feature hooks and sections own feature-specific client logic.
- Domain logic belongs in named feature/domain modules, not generic helpers.

Current intentional compatibility shims

- `src/bee_ingestion/api.py` remains a compatibility entrypoint while route ownership lives under `http_api/routes/`.
- `src/bee_ingestion/repository.py` remains a compatibility facade while bounded stores replace broad persistence ownership.
- `src/bee_ingestion/service.py` remains a worker-facing facade while offline stage bodies move into `offline_pipeline/stages/`.
- `src/bee_ingestion/agent.py` remains the single orchestrator facade while prompt and verifier responsibilities live in `agent_runtime/`.

Known remaining hotspots

- `src/bee_ingestion/repository.py` still owns document/chunk/KG/job persistence in one module.
- `src/bee_ingestion/service.py` still owns replay, repair, reindex, and maintenance workflow bodies inline.
- `frontend/src/features/admin/AdminExtendedSections.tsx` is still larger than ideal.
- `src/bee_ingestion/http_api/dependencies.py` is an intentional proxy seam for tests and lazy initialization, but it should not grow into a behavior-owning module.

Success criteria for future architecture passes

- `api.py` stays a thin compatibility entrypoint.
- Admin route modules stay thin and call admin services.
- Bounded stores, not `repository.py`, own extracted persistence SQL.
- `service.py` becomes primarily a worker/job facade plus pipeline runner wiring.
- Retrieval remains in-process.
- Ingestion remains offline/job-oriented.
- Platform logic and hive-domain logic stay clearly separated.
